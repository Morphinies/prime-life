# Архитектура Frontend

## Назначение документа

Этот документ описывает архитектуру `frontend/src` и правила разработки frontend-части проекта на основе Feature-Sliced Design (FSD).

Цель архитектуры:

- держать код предсказуемым
- упрощать навигацию по проекту
- снижать связанность между частями приложения
- упрощать развитие и рефакторинг
- не допускать хаотичного роста кода по мере масштабирования проекта

Этот документ нужно использовать как практическое руководство при добавлении новых экранов, фич, сущностей, виджетов и общих утилит.

## Базовый принцип FSD

Frontend делится на слои. Верхние слои могут зависеть от нижних. Нижние от верхних зависеть не должны.

Направление зависимостей:

`app -> pages -> widgets -> features -> entities -> shared`

Это означает:

- `pages` могут использовать `widgets`, `features`, `entities`, `shared`
- `widgets` могут использовать `features`, `entities`, `shared`
- `features` могут использовать `entities`, `shared`
- `entities` могут использовать `shared`
- `shared` не должен зависеть ни от одного бизнес-слоя

Запрещённые зависимости:

- `shared -> entities/features/widgets/pages/app`
- `entities -> features/widgets/pages/app`
- `features -> widgets/pages/app`
- `widgets -> pages/app`

Слой `app` является корнем композиции. Он инициализирует приложение, провайдеры, глобальный store, глобальные стили и общую конфигурацию.

## Структура frontend/src

Текущая структура:

- `app`
- `assets`
- `entities`
- `features`
- `pages`
- `processes`
- `shared`
- `widgets`

Ниже описано назначение каждого слоя.

## Слой app

### Назначение

Слой `app` отвечает за:

- инициализацию приложения
- подключение провайдеров
- настройку глобального store
- подключение глобальных стилей
- корневую конфигурацию приложения

### Что сюда относится

Примеры из проекта:

- `app/providers`
- `app/store`
- `app/styles`

### Правила

- `app` может импортировать любой нижележащий слой
- другие слои не должны импортировать `app`
- всё, что относится к глобальному состоянию приложения, должно находиться здесь

### Примеры подходящего кода

- настройка Redux store
- настройка React Query provider
- настройка Ant Design provider
- инициализация темы
- глобальные обёртки приложения

## Слой pages

### Назначение

Слой `pages` отвечает за:

- роутовый уровень композиции
- сборку страницы из виджетов и фич
- page-specific layout
- loader/client-loader логику

### Что сюда относится

Примеры из проекта:

- `pages/dashboard`
- `pages/projects`
- `pages/tasks`

### Правила

- страницы должны быть тонкими
- страница должна собирать экран из нижних слоёв, а не хранить в себе сложную бизнес-логику
- page-specific UI и page-specific content допустимы, если они нужны только этой странице

### Хорошие примеры

- `pages/tasks/index.tsx` как точка композиции страницы
- `pages/tasks/ui/HeadController.tsx` как page-specific UI

### Плохие примеры

- CRUD-логика сущности прямо внутри page-компонента
- создание переиспользуемых UI-примитивов внутри `pages`

## Слой widgets

### Назначение

Слой `widgets` отвечает за:

- крупные самостоятельные UI-блоки
- композицию нескольких lower-level модулей
- блоки интерфейса, которые можно использовать на разных страницах

### Что сюда относится

Примеры из проекта:

- `widgets/TaskList`
- `widgets/Header`
- `widgets/Sidebar`
- `widgets/Layouts`

### Правила

- widget может содержать orchestration-логику
- widget может использовать `features`, `entities`, `shared`
- widget не должен импортировать `app`
- widget не должен зависеть от конкретной страницы
- внутренние части widget должны лежать рядом с ним

### Когда файл должен оставаться внутри widget

- он используется только этим widget
- он не является общей доменной сущностью
- он не является самостоятельной user-action feature

### Хорошие примеры

- `widgets/TaskList/index.tsx`
- `widgets/TaskList/useTaskListController.ts`
- `widgets/TaskList/ModalTask.tsx`

## Слой features

### Назначение

Слой `features` отвечает за:

- пользовательские действия
- пользовательские сценарии
- прикладные действия над данными

### Что сюда относится

Примеры из проекта:

- `features/auth`
- `features/theme`

### Правила

- feature появляется там, где есть самостоятельное действие пользователя
- feature может использовать `entities` и `shared`
- feature не должна существовать только как формальная прослойка без собственной ответственности

### Примеры того, что хорошо ложится в features

- login
- register
- toggle theme
- create project
- invite user
- task filtering

### Когда не нужно создавать feature

- если модуль просто проксирует существующий entity hook без добавочной логики
- если новая папка только усложняет навигацию по проекту
- если ответственность по факту принадлежит `entities` или `widgets`

## Слой entities

### Назначение

Слой `entities` отвечает за:

- доменные сущности
- типы сущностей
- API сущностей
- model-логику сущностей
- UI одной сущности

### Что сюда относится

Примеры из проекта:

- `entities/task`
- `entities/user`

### Внутренняя структура entity slice

Обычно:

- `api`
- `model`
- `ui`
- `index.ts`

### Правила

- entity должна представлять бизнес-сущность
- entity UI должна отображать сущность и принимать действия через props
- entity не должна знать о роутинге, страницах, app store или page composition
- entity может зависеть только от `shared`

### Хорошие примеры

- `entities/task/model/types.ts`
- `entities/task/api/task-queries.ts`
- `entities/task/ui/TaskCard.tsx`

## Слой shared

### Назначение

Слой `shared` отвечает за:

- общие технические примитивы
- переиспользуемые UI-компоненты без бизнес-смысла
- низкоуровневые хелперы
- конфигурацию, не привязанную к конкретной сущности
- адаптеры библиотек

### Что сюда относится

Примеры из проекта:

- `shared/api`
- `shared/config`
- `shared/lib`
- `shared/ui`
- `shared/utils`

### Правила

- `shared` это самый нижний слой
- `shared` не должен зависеть от `entities`, `features`, `widgets`, `pages`, `app`
- `shared` не должен содержать доменную семантику конкретной сущности

### Хорошие примеры

- `shared/ui/Icon`
- `shared/ui/Substrate`
- `shared/api/axios-instance.ts`
- `shared/config/colors.ts`

### Плохие примеры

- импорт типов `Task` в `shared/config`
- чтение app store из `shared/lib`

## Слой assets

### Назначение

Слой `assets` отвечает за:

- иконки
- изображения
- статические ресурсы

### Правила

- `assets` не определяет архитектурные границы
- в assets не должна прятаться прикладная логика

## Слой processes

### Назначение

Слой `processes` предназначен для:

- длинных пользовательских процессов
- сценариев, охватывающих несколько страниц
- сложных прикладных flow

### Текущее состояние

Сейчас слой пустой.

Его не нужно заполнять заранее. Использовать `processes` имеет смысл только тогда, когда действительно появляется межстраничный или длинный сценарий.

## Что такое slice

Slice это отдельная единица ответственности внутри слоя.

Примеры:

- `entities/task`
- `features/auth`
- `widgets/TaskList`
- `pages/tasks`

Slice должен иметь понятную границу ответственности. Если граница размыта, значит структура выбрана неудачно.

## Рекомендуемая внутренняя структура slice

### Для entities

- `api`
- `model`
- `ui`
- `index.ts`

### Для features

- `api` при необходимости
- `model`
- `ui`
- `index.ts`

### Для widgets

- `index.tsx`
- локальные subcomponents
- локальные hooks/controllers

### Для pages

- `index.tsx`
- `ui`
- `content.ts` при необходимости

## Правило public API

Если slice используется снаружи, он должен предоставлять внешний API через `index.ts`.

### Зачем это нужно

- скрывает внутреннюю структуру
- уменьшает количество хрупких deep imports
- упрощает рефакторинг
- делает зависимости между срезами явными

### Примеры

- `entities/task/index.ts`
- `features/theme/index.ts`

### Нужно избегать

- глубоких импортов во внутренние файлы другого slice без необходимости

### Допустимое исключение

- локальные импорты внутри одного и того же slice

## Правила взаимодействия между слоями

## app и lower layers

`app` хранит глобальный store, провайдеры и инициализацию.

Если lower layer нужно поведение, завязанное на глобальное состояние, это поведение должно быть вынесено в подходящую нижележащую абстракцию, а не тянуть прямые импорты из `app`.

Пример:

- переключение темы оформлено через `features/theme`
- widget использует `features/theme`
- widget не импортирует напрямую Redux actions/selectors из `app`

## widgets и entities

Widget может использовать entity hooks и entity UI.

Пример:

- `widgets/TaskList` использует `entities/task`

Но при этом:

- widget-specific orchestration не должна уходить в entity UI
- page-specific layout не должен уходить в entity UI
- entity UI должна оставаться переиспользуемой

## shared и бизнес-слои

`shared` обслуживает все остальные слои.

Если модуль в `shared` начинает зависеть от `task`, `user`, `project` или других бизнес-сущностей, его место, скорее всего, не в `shared`.

Такой код надо переносить в:

- `entities`
- `features`
- реже в `widgets`

в зависимости от реальной ответственности.

## Текущие архитектурные решения проекта

## TaskList

`TaskList` это widget.

Почему:

- он загружает и синхронизирует список
- управляет модалкой
- выполняет CRUD и reorder orchestration
- композиционно использует entity UI

Текущее место:

- `widgets/TaskList`

## TaskCard

`TaskCard` это entity UI.

Почему:

- она представляет одну задачу
- умеет отрисовать сущность
- действия получает через props
- не содержит page-level или widget-level композиции

Текущее место:

- `entities/task/ui/TaskCard.tsx`

## ModalTask

`ModalTask` сейчас остаётся внутри `widgets/TaskList`.

Почему:

- она используется только `TaskList`
- это часть внутренней композиции widget
- она пока не является отдельной переиспользуемой feature

Если в будущем редактирование задачи станет отдельным самостоятельным сценарием, надо будет рассмотреть перенос в:

- `features/task-edit`
- либо в `entities/task/ui` как reusable task form/editor

## Theme

Работа с темой оформлена через `features/theme`.

Почему:

- переключение темы это пользовательское действие
- widgets и pages не должны импортировать app store напрямую

`shared/lib/hooks/useThemeToken` ограничен доступом только к Ant Design theme tokens и не содержит app-specific state logic.

## Правила именования

Общие правила:

- использовать имена по ответственности, а не по техническому признаку
- избегать слишком общих названий
- выбирать имя, которое сразу объясняет роль модуля

### Хорошие примеры

- `TaskList`
- `useTaskListController`
- `ThemeToggleButton`
- `task-queries`

### Плохие примеры

- `useTasks`, если hook управляет и списком, и модалкой, и CRUD-сценариями
- `helpers2`
- `commonThing`
- `dataUtils`

## Когда создавать новый slice

Создавать новый slice стоит, если выполняется хотя бы одно условие:

- появляется новая бизнес-сущность
- появляется новый пользовательский сценарий
- модуль начинает переиспользоваться в нескольких местах
- у логики появляется отдельная стабильная ответственность

Не нужно создавать новый slice, если:

- это формальная прослойка без собственной ценности
- это просто тонкий proxy над уже существующим кодом
- структура становится сложнее, а связанность не уменьшается

## Правила масштабируемой разработки

Чтобы frontend оставался масштабируемым:

- держать страницы тонкими
- размещать логику рядом с её реальным владельцем
- не превращать `shared` в склад бизнес-логики
- использовать public API slices
- не допускать обратных зависимостей вверх по слоям
- предпочитать композицию вместо монолитных компонентов
- исправлять архитектурные нарушения сразу, пока они локальны

## Чеклист перед добавлением нового кода

Перед тем как добавить новый файл, нужно проверить:

1. Какой слой действительно владеет этой ответственностью?
2. Зависит ли файл только от допустимых lower layers?
3. Это user action, domain model, page composition, widget composition или primitive?
4. Это должно быть внутренней частью slice или частью public API?
5. Не создаётся ли случайная зависимость от `app`, конкретной страницы или другой бизнес-сущности?
6. Будет ли другому разработчику понятно, где искать и расширять этот код?

## Антипаттерны

Нужно избегать следующего:

- хранить доменные типы в `shared`
- импортировать app store selectors/actions в `entities`, `features` или `widgets` напрямую
- складывать крупную бизнес-логику в `pages`
- хранить generic UI-примитивы в `widgets`
- создавать пустые или формальные слои без ответственности
- использовать deep imports во внутренности чужого slice без необходимости

## Сигналы к рефакторингу

Нужно пересматривать структуру, если:

- widget начинает использоваться как generic primitive
- локальный компонент становится самостоятельным сценарием
- shared helper начинает тянуть доменные зависимости
- entity UI начинает знать о page/widget orchestration
- страница становится слишком тяжёлой
- feature превращается в пустую обёртку

## Практическое правило

Если возникают сомнения, куда положить код, ориентироваться нужно не на тип файла, а на ответственность.

То есть:

- не каждый hook должен жить в `shared`
- не каждый компонент должен жить в `ui`
- не каждая модалка является feature
- не каждый API hook требует отдельного feature slice

Главный вопрос:

`Кто владеет этим поведением?`

Ответ на этот вопрос определяет слой и slice.
